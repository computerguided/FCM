FCM in Embedded Software Architecture

A method for structuring embedded systems as Devices of cooperating Functional Components and Handlers — where asynchrony from the world is converted into synchronous, deterministic message handling inside a strict run-to-completion boundary.

01Introduction

Embedded software lives at the meeting point of computation and the physical world. It runs on resource-constrained targets, drives sensors and actuators, talks to peripherals over buses with their own timing, exchanges packets over wireless and wired links, and must remain correct under interrupts, concurrent events, and hard timing constraints. The defining architectural challenge is not the algorithms themselves but managing concurrency, asynchrony, and state safely across a system that must always know what it is doing.

The Functional Components Method (FCM) is a software architecture method specifically aimed at this challenge. It structures an embedded system as a Device — a collection of Functional Components and Handlers working together to perform a specific function — with each Functional Component described as a state machine, and with a strict separation between the synchronous inner world of the application logic and everything that cannot fit inside it: asynchronous hardware events, driver callbacks, network traffic, and long-running computations that would otherwise stall the message loop.

Stadiæ is a graphical editor built around FCM. Where general-purpose diagram tools treat state machines as drawings, Stadiæ treats the FCM concepts themselves — Devices, Functional Components, Handlers, Interfaces, Messages, type definitions, state machines with their pseudostates, and per-Component state variables, constants, and local functions — as first-class entities, with the two FCM levels (the Device diagram and each Component's state machine) sharing one vocabulary and one underlying model. The editor is therefore not just for drawing FCM diagrams but for authoring an FCM specification: the same model drives the rendered diagrams, the JSON save format, and a full specification export to HTML or Word.

This document is the conceptual reference for the method itself. It explains FCM in its own terms in the order best for a first read: the Device and its boundary, the two levels at which FCM describes the Device, the entities inside it (Functional Components, Interfaces, Handlers) and the run-to-completion discipline that governs them, then how a Functional Component's behaviour is described, then how an architect produces this whole structure from requirements. Readers wanting a how-to first — how an architect finds Functional Components and Handlers from a set of requirements — can read §9 immediately after §1 without losing the thread.

02The Device

In FCM, the Device is the architectural unit being designed. A Device can be a physical entity (an embedded controller, a network node, a sensor module) or a software entity (an application, a service); what matters at the architecture level is that it is a self-contained collection of cooperating Functional Components and Handlers, with one shared message queue and a defined boundary against the outside world.

Inside the Device boundary live the application logic that the team owns and reasons about: the Functional Components, the Handlers that shield the rest of the Device from anything that cannot be done synchronously inside an action — there are two kinds, Asynchronous Interface Handlers for asynchronous events from the environment and Worker Handlers for long-running computational work — and the Interfaces over which all of these communicate. Outside the boundary lies everything the Device must deal with but does not control — the MCU's interrupt sources, the radio stack, the display controller, the sensor bus, the operating system or RTOS scheduler, the host network.

This boundary is not a packaging detail. It is the line across which asynchrony from the environment is converted into synchronous, deterministic message handling inside the Device. Drawing this boundary explicitly is the first architectural act of an FCM design.

A Device carries a free-text specification: the design intent and constraints (functional requirements, timing budgets, safety invariants, resource limits) that all Functional Components and Handlers must conform to. It is the contract the Device upholds against the rest of the product.

The Device also owns the infrastructure that makes the FCM model operational: a Message Queue through which every message flows, a Timer Handler singleton that converts time into Timeout messages, and the construction and connection of every Component and Handler at start-up. These elements are part of the FCM runtime model rather than something the architect specifies per-Device — but they are real architectural artefacts that the system specification implicitly relies on.

03The Two Levels of Description

FCM describes an embedded software architecture at two levels, which together form the complete specification:

  • The Device level shows the decomposition: which Functional Components and Handlers exist, which Interfaces they speak, which type definitions ground the data contract, and which Functional Components depend on which Handler APIs. It answers: what is the Device made of, where is the boundary with the hardware and platform, and how do the parts connect? This is the level at which integration, hardware-abstraction strategy, and inter-component contracts are reviewed.
  • The Functional Component level shows the behaviour: each Functional Component's state machine and its state transition table, drawn against the shared interface and message vocabulary, complemented by its state variables, constants, and local functions. It answers: given this Functional Component, how does it behave when its messages arrive? This is the level at which functional correctness, mode logic, timing behaviour, and fault handling are reviewed.

The two levels share their vocabulary — the same interfaces, messages, type definitions, Functional Components, and Handlers appear at both. This shared vocabulary is what holds the architecture together as a single design rather than two parallel ones; what that buys the embedded architect concretely is taken up again in the closing section.

04Functional Components

A Functional Component (often shortened to Component when context is unambiguous) is the primary FCM building block: a self-contained unit of behaviour described by a state machine. A Functional Component typically corresponds to a logical responsibility — a connection manager, a peripheral controller, a session, an application-level feature — rather than to a hardware block or a thread.

Functional Components are strictly encapsulated:

  • A Functional Component's states, choice-points, and transitions are entirely its own — no other Functional Component can refer to them.
  • Functional Components communicate only through messages on shared interfaces, never by calling each other directly and never by sharing variables.
  • A Functional Component has no visibility into another Functional Component's state.

This encapsulation is the heart of FCM: behaviour is local, vocabulary is shared. In embedded terms, this gives the architectural properties teams actually need:

  • Each Functional Component can be unit-tested in isolation by stimulating its message interfaces — no hardware required.
  • A Functional Component can be re-implemented (e.g. when a peripheral is swapped, or for a new product variant) without touching its peers, as long as the interface contract is preserved.
  • Concurrency reasoning is bounded: there are no shared-state hazards between Functional Components by construction, only well-defined messages on the queue.

A Functional Component is fully documented by its state machine plus three documentation-only descriptors: the state variables it holds, the constants it references, and the local functions that capture reusable steps within its actions.

05Interfaces, Messages, and Type Definitions

An Interface is a contract between Functional Components (and between Components and Handlers). It groups a related set of messages under a name (e.g. Connection owns ConnectReq, ConnectedInd, DisconnectedInd). Interfaces are shared: when two Functional Components both speak Connection, they speak the same Connection. An interface is therefore not a property of a Component but a vocabulary that exists at the Device level — the Device-wide protocol dictionary.

A Message is the unit of communication. It triggers a transition in the receiving Functional Component's state machine, and it can carry typed parameters (name + type + description) that supply the data the transition needs. The set of messages on an interface, together with their parameter types, is the full data contract — the equivalent, at the architecture level, of a header file or an IDL definition for that interface.

To keep this contract precise, FCM also supports type definitions: Device-wide named domain types (UserId, Timestamp, Currency, PaymentState) referenced from message parameters and from Functional-Component state-variable types. Type definitions give the architecture a shared vocabulary for what data flows through the Device, complementing the interface vocabulary that captures what events happen.

Two intrinsic interfaces

Two interfaces are conceptually intrinsic to FCM and always present:

  • Timer — carries the Timeout message. In embedded software, timing is pervasive (debounce intervals, retry backoffs, watchdog deadlines, protocol timers). FCM brings these into the same vocabulary as everything else: a timeout is just a message that arrives in the queue, handled by exactly the same mechanism as any other event. The Device-level Timer Handler is the singleton that produces these messages, mediating between the platform's timer hardware or OS scheduler and the synchronous Device.
  • Logical — every Functional Component has an implicit Logical interface that carries Yes and No messages. These are emitted by choice-point evaluation functions and consumed by the transitions out of that choice-point. Logical is purely internal to the Functional Component and never crosses interface wiring.

06Handlers and the Run-to-Completion Boundary

This section captures what is arguably the most important architectural decision in any embedded system, and the one FCM is most explicit about.

6.1The synchronous inner world: run-to-completion

Inside the Device, message-passing is synchronous and serialised, governed by a strict run-to-completion (RTC) semantics and mediated by a single Device-wide Message Queue. Every message — whether produced by a Functional Component, an Asynchronous Interface Handler, a Worker Handler, or the Timer Handler — flows through this one queue, and the Device dispatches messages from it one at a time.

When a message is dispatched to a Functional Component, three things happen as one indivisible step:

  1. The Functional Component's state transition table — the mapping from (current state, interface, message) to (action, target state) — selects the transition that the current state and the incoming message identify.
  2. The transition's action runs to completion — every step in the action, every Handler function call, every state-variable update, every local-function invocation — without interruption from other messages.
  3. The Functional Component arrives at the target state.

Only when this entire sequence has completed does the dispatcher pick up the next message from the queue. No message is ever processed concurrently with another. No Functional Component is ever caught between states. No action is ever observed half-executed.

The reasoning model is simply: given this state and this incoming message, the Functional Component runs exactly this action to completion and arrives at exactly this state.

This run-to-completion contract is what gives "synchronous" its operational meaning. It is the architectural guarantee that:

  • State is always consistent. A Functional Component never has to guard against being observed mid-transition, because by definition no observation can occur mid-transition.
  • Actions are atomic with respect to the model. Whatever an action does — drive a peripheral, update three state variables, request a timer, emit a follow-up message — is either fully done or fully not done.
  • Reasoning is sequential. The architect can read an action top to bottom and trust that nothing else will interleave with it.
  • Re-entrancy is impossible by construction. A Functional Component cannot receive a new message while still handling the previous one.

RTC implies a discipline on action design: actions must be bounded in execution time and must not block. The FCM idiom is therefore: when a Functional Component needs to wait for something, it does not block in the action — it calls a Handler function to initiate the wait, transitions to a state that represents the wait, and lets the eventual completion arrive as a new message that triggers the next transition.

This synchronous, run-to-completion discipline is what makes FCM tractable for embedded software. It eliminates the failure modes that dominate naive embedded designs:

  • No race conditions between event handlers and main-loop code.
  • No re-entrancy hazards when a callback fires mid-update.
  • No "what state were we in when the interrupt happened?" debugging sessions.
  • No locks, no critical sections sprinkled through application logic, no priority-inversion surprises.

6.2Why work has to leave the inner world

Two distinct kinds of work cannot fit inside the run-to-completion step of a transition action — and they are exactly the two kinds of boundary analysis identified as Handler candidates.

The first is the asynchronous outside world, which is aggressively asynchronous in embedded software: hardware interrupts fire on their own schedule (UART RX, GPIO edges, DMA completions), radio stacks deliver packets when the link delivers them, peripheral drivers report completions via callbacks from arbitrary contexts, timers expire independently of the application's main flow, higher-priority tasks preempt at any instruction boundary, and user input arrives whenever the user feels like it. If those raw asynchronous events were allowed to reach Functional Components directly, both the synchronous-handling guarantee and the run-to-completion guarantee would collapse.

The second is long-running internal computation — image decoding, cryptographic operations, signal-processing pipelines, search and planning algorithms. These are deterministic computations with no asynchronous edge, but performing one of them inside an action would stall the entire Device's message loop for the duration of the computation, miss timer deadlines, drop hardware events, and break every timing guarantee the architecture depends on.

FCM addresses both with the same kind of construct: a Handler that sits on the Device boundary and bridges the synchronous inner world to whatever cannot happen synchronously inside an action. The two kinds — Asynchronous Interface Handler and Worker Handler — are peers, not parent and specialisation: they answer different architectural questions and are identified at different points in boundary analysis. What they share is structural: both have no state machine, both expose synchronous non-blocking functions to Functional Components, and both deliver results back as messages on the Device's Message Queue.

6.3Asynchronous Interface Handlers

An Asynchronous Interface Handler (AIH) translates between the asynchronous environment and the synchronous Device, in two complementary directions:

Inbound: asynchronous events → synchronous messages. The AIH owns the asynchronous machinery for some external concern — an ISR, a radio callback, a peripheral driver, a queue from another task or core. When an outside event occurs (a byte on the UART, a BLE notification, a DMA completion, an OS signal), the AIH captures it in whatever execution context it must, and posts a well-defined message to the Device's Message Queue on a shared interface.

Outbound: synchronous calls → asynchronous operations. When a Functional Component needs to act on the outside world — drive a GPIO, transmit a frame, write a flash sector, request a sensor reading — it calls a function that the AIH exposes. The call returns immediately and the Functional Component continues its action. The AIH then performs the actual asynchronous work in whatever way the platform requires (DMA, scheduled task, kernel call), and any result of that work eventually returns as a message — translated back into the synchronous inner world by the same AIH.

This split is fundamental to embedded architecture:

  • Asynchronous events come in as messages. A Functional Component reacts to events; it never spins, polls, or blocks waiting for them.
  • Synchronous calls go out as functions. A Functional Component requests work and returns. It does not wait for the result; the result will come back later as another message, possibly in a different state.

It is also why an AIH exposes a callable API of functions with typed parameters rather than a state machine: an AIH's outward face is a set of synchronous, non-blocking entry points; its asynchronous internals are deliberately outside the FCM model.

6.4Worker Handlers

A Worker Handler addresses a different problem: a piece of computation that the Device must perform but that cannot fit inside the run-to-completion step of any single action — image decoding, cryptographic operations, signal processing, complex search or planning, anything that takes long enough that performing it synchronously would stall the Device's message loop.

Architecturally, a Worker Handler exposes the same kind of outward face as an AIH — synchronous, non-blocking entry points to start, cancel, or parameterise the work, and a result delivered as a message on the Device's Message Queue. From a Functional Component's perspective, the two kinds of Handler are used identically: call a function to initiate work, transition to a state that represents the wait, and react to the completion message when it arrives. Internally, however, a Worker Handler runs its work in its own thread, with cooperative cancellation so the Device can ask the worker to stop and have it actually do so.

6.5The Timer Handler

The default Timer interface is the canonical example of the synchronous/asynchronous discipline. Hardware timers and OS scheduling are asynchronous; in FCM, time arrives as just another message. The Device provides a built-in Timer Handler singleton that owns the platform's timer machinery and emits Timeout messages onto the Message Queue, dispatched under the same run-to-completion rules as everything else. From a Functional Component's perspective, time is simply another event source — request a timer, transition to a waiting state, and react when the Timeout arrives.

07The State Machine

A Functional Component's behaviour is fully described by its state machine, captured architecturally as a state transition table: a mapping from (current state, interface, message) to (action, target state). The transition table is the canonical FCM artefact — every behavioural decision the Functional Component makes is recorded as a row in this table, which makes the design exhaustively reviewable.

State machines are the right abstraction for embedded software because so much of what embedded systems do is mode-dependent: a connection manager behaves differently when idle, advertising, connecting, or operational; a sensor sequencer behaves differently across initialisation, calibration, sampling, and standby.

A Functional Component's state machine consists of:

  • States — the distinct situations the Functional Component can be in. A useful naming guideline is that a state should answer the question "what is the Functional Component doing?": Advertising, Connecting, Operational, LowPower all read naturally as I am Advertising, I am Connecting. At any moment the Functional Component is in exactly one state.
  • Choice-points — decision nodes where the Functional Component asks a yes/no question (e.g. "Is Server Whitelisted?", "Battery above threshold?"). On entry to a choice-point, an evaluation function runs against the Functional Component's state variables and returns true or false. The result is immediately translated into a Yes or No message on the Functional Component's implicit Logical interface. A choice-point is not a state in which the Functional Component rests; it is a point through which the Functional Component passes while choosing a path.
  • Transitions — the rows of the state transition table. A transition has a source, a target, and one or more triggering messages, and it carries an action. When a triggering message arrives while the Functional Component is in the source state, the transition fires: its action is executed under run-to-completion and the Functional Component arrives at the target.

The three pseudostates

Three pseudostates complete the FCM semantics, each addressing a recurring embedded need:

  • START ● — the Functional Component's first state when the Device is initialised. By convention, the first state in the state list is treated as the initial state; the architecture-level diagram makes it explicit so the reader knows where the Functional Component starts.
  • History H — meaning "re-enter the state that was last active". Used when the Functional Component should resume where it was, for example after handling an interruption such as a low-power excursion, an over-temperature event, or a transient peripheral failure.
  • Any ∗ — the wildcard source, meaning "from any state". Used for cross-cutting embedded behaviours that apply regardless of where the Functional Component currently is — Reset, PowerOff, FaultDetected. A given message can drive at most one Any-source transition. Any can be combined with History (∗ → [H] = "from any state, return to where we were"), a common idiom for handling transient faults.

Any is a sharp tool: if a Functional Component grows several Any-source transitions for ordinary messages, it is usually a sign that real state structure is missing — that the messages in question should be reachable only from specific states, and the design is hiding behind the wildcard rather than committing to which states actually accept them.

08State Variables, Constants, and Local Functions

Three further per-Component concepts complete the description of a Functional Component's behaviour. They are documentation-only at the architecture level — they do not appear on the diagram — but together they turn the state machine from a purely diagrammatic artefact into a complete behavioural specification.

State variables

State variables describe the data the Functional Component holds — counters, identifiers, cached configuration, accumulated measurements, retry counts, references to outstanding requests. They make explicit what the Functional Component remembers in addition to its current state. They are read inside choice-point evaluation functions and read or written inside transition actions. Naming and typing them at the architecture level means RAM usage is visible in the design, not discovered late during integration. Under run-to-completion, state variables are accessed exclusively within actions and evaluation functions — and because both run atomically as part of message dispatch, there is never a question of who is updating them concurrently.

Constants

Constants are named values that the Functional Component uses but does not change at runtime — a RetryLimit, an AdvertisementInterval, a MaxPayloadSize, a magic protocol identifier. They are referenced by name from action text so the parameter that drives behaviour is documented in the architecture rather than hidden as a literal in the implementation. Separating constants from state variables is itself architectural information: it makes clear which values are configuration (potentially tunable at integration or build time) and which are runtime states.

Local functions

Local functions capture sequences of steps that recur across multiple transition actions — the typical "reset all peripherals", "flush the buffer", "log and notify" routines that would otherwise be copied into many actions. Naming them once and referring to them by name from the actions that need them avoids duplication and gives a vocabulary for the Functional Component's internal operations. Local functions take no parameters; they operate on the Functional Component's state variables and constants directly. Like the actions that invoke them, they execute as part of the run-to-completion step of the transition that called them.

Together, state variables, constants, and local functions complete the Functional Component specification: what it remembers, what it is configured with, and what reusable internal operations it performs.

09From Requirements to Components

Before any state machine is drawn, FCM expects the architect to determine what should be a Functional Component, what should be a Handler, and what lies outside the Device altogether. Functional Components do not appear by intuition or by mirroring the hardware block diagram; they emerge from a deliberate two-step analysis: functional decomposition of the requirements and boundary analysis of the environment.

9.1Boundary analysis: identifying Handlers first

The most reliable starting point is the Device's edge, because the environment is concrete: the hardware datasheet, the wireless protocol stack, the host operating system, the user interface peripherals, and the computational workloads the Device must perform without blocking its message loop.

FCM provides two kinds of Handler, each addressing a distinct reason why work cannot happen synchronously inside a Functional Component's action.

Asynchronous Interface Handlers (AIHs) mediate every distinct asynchronous source or sink in the environment:

  • A radio or network stack that delivers incoming traffic asynchronously and accepts outgoing traffic via a synchronous API.
  • A peripheral driver (display controller, audio codec, sensor) that signals events through callbacks and exposes commands through function calls.
  • A storage subsystem (flash, EEPROM, file system) where reads and writes complete asynchronously.
  • An OS or RTOS service whose notifications arrive on signals or queues from outside the Device's main context.
  • A timer source — though FCM provides this as a built-in Timer Handler.

Worker Handlers mediate long-running computational tasks that the Device must perform but that cannot fit inside the run-to-completion message loop — image decoding, cryptographic operations, signal processing, complex search or planning algorithms. The decisive question is not asynchrony of an external event but execution time of internal work: any computation that "would take too long to do inside an action" is a candidate Worker Handler.

Listing the Handlers of both kinds fixes the Device boundary precisely: anything those Handlers translate to and from is inside the Device; their internal asynchronous or threaded machinery is outside. This step alone often uncovers requirements that were ambiguous at the textual level: who delivers this event? who initiates this action? is this thing a peripheral we drive or a service we react to? is this computation fast enough to live inside an action, or does it need a Worker?

9.2Functional decomposition: identifying Components

With the boundary established, the requirements are decomposed into cohesive units of behaviour that will become Functional Components. Each candidate Functional Component is examined against three criteria:

  • Single responsibility. The Functional Component owns one well-defined concern — a connection lifecycle, a session, a feature, a control loop, a sequencer. A useful test: it should be possible to summarise the Functional Component's purpose in one sentence without using "and".
  • Mode-driven behaviour. The Functional Component's responsibility involves distinct modes or phases that justify a state machine — there is a meaningful difference between what it does when idle and what it does when active. Pure data-transformation logic with no state typically belongs inside a Functional Component's actions or local functions.
  • Independent lifecycle. The Functional Component can start, stop, recover, or fail without dragging the rest of the Device along. If two candidate Functional Components are always in the same mode at the same time, they are probably one Functional Component.

This typically produces a layered set of Functional Components: lower-level ones that mediate between Handlers and higher-level logic (e.g. a connection manager, a peripheral controller), and higher-level ones that orchestrate features and user-visible behaviour. The layering is logical, not enforced — Functional Components are peers in the architecture, all communicating through the same kind of interface.

9.3Identifying interfaces and messages

Interfaces emerge from the interactions between the units identified above. For each pair of Functional Components — and for each Component-Handler pair on the boundary — the architect asks:

  • What does one need to tell the other? These become messages.
  • What logical contract groups those messages? That is the interface.

A good interface is a coherent vocabulary, not a bag of unrelated messages. Messages on a Connection interface all concern the lifecycle of a connection; messages on a Content interface all concern content delivery. When messages on one interface seem to fall into two unrelated groups, that is usually a sign the interface should be split.

9.4Iteration: the architecture converges

The first pass of this analysis is rarely the last. FCM supports a sketch, then formalise workflow: an initial decomposition is captured, the system diagram is drawn, and problems become visible:

  • A Functional Component that touches an unreasonable number of interfaces probably has more than one responsibility — split it along the natural seam.
  • A Functional Component that touches only one interface and does nothing but forward messages is probably not a Functional Component — fold its logic into the neighbour.
  • Two Functional Components that need to share state suggest a missing Functional Component — extract the shared state into a new Functional Component.
  • A Handler that translates events into messages no Functional Component cares about is either premature or its translation is too fine-grained.
  • An interface spoken by only one Functional Component (other than its Handler) suggests that interface is really an internal concern — demote it.
The decomposition converges when each Functional Component has a clear single responsibility, each interface has a coherent purpose, each Handler corresponds to a real environmental concern, and the Device diagram reads cleanly to someone who has never seen the design before. The architecture is right when it explains itself.

10Coherence

The architectural value FCM delivers comes from one structural property running through everything above: the Device level and the Functional Component level share a single vocabulary. The same interfaces, messages, type definitions, Functional Components, and Handlers appear at both, and the same run-to-completion discipline governs how they interact.

A change to an interface is visible everywhere it is spoken; a Functional Component's behaviour can be read against the same message names that appear on the Device diagram; the data contract is grounded in the same type definitions everywhere they are referenced. For embedded software — where mistakes found late are expensive and on-target debugging is painful — this coherence is what allows the architecture to be reasoned about, reviewed, and modified as a single design rather than a stack of overlapping documents.
